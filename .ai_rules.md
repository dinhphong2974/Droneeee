# QUA TẮC BẮT BUỘC DÀNH CHO AI (DƯỚI 20 DÒNG)

1. **Đồng bộ GCS/ESP và INAV**: Mỗi khi thay đổi code hệ thống GCS/ESP mạch chính mà kéo theo việc cần thay đổi một dòng lệnh CLI trong INAV (băng thông, cổng UART, Protocol, Failsafe...), BẮT BUỘC phải thông báo rõ ràng lệnh CLI đó cho người dùng.
2. **Review & Test**: Mỗi khi đưa ra bất kỳ thay đổi code nào, BẮT BUỘC phải review lại tổng thể, tự soát lỗi (debug) và kiểm tra tính hợp lý (test logic) với toàn bộ project. Không được code bừa gây hỏng chức năng đang chạy.
3. **Câu chốt hạ**: Cuối mỗi lượt trả lời, BẮT BUỘC phải luôn thêm đúng nguyên văn câu này: "Tôi đã xem file rules để tôi chắc chắn rằng bạn không quên các rules khác".
4. **INAV Quirks & Firmware Versions**: Khi đọc/đánh giá CLI của thiết bị INAV, BẮT BUỘC phải luôn đọc chiếu file `config/INAV_QUIRKS.md` để đối soát sự thay đổi về Mode IDs (BoxID) giữa các firmware/fork khác nhau (như SpeedyBee 9.x vs bản chuẩn), tuyệt đối không phán đoán BoxID theo kinh nghiệm cũ gây báo lỗi sai định dạng hệ thống.
5. **Phân tích trước — Code sau**: Mỗi khi người dùng yêu cầu làm bất kỳ thay đổi nào, BẮT BUỘC phải: (a) Đưa ra nhận xét/phân tích vấn đề, (b) Đề xuất phương án giải quyết, (c) Lập plan chi tiết, (d) CHỜ người dùng xác nhận rồi MỚI ĐƯỢC sửa code. Tuyệt đối không tự ý sửa code trước khi được duyệt.
